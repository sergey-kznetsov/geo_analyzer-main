# Проверка ключа 2GIS

Перед полным анализом можно проверить активный ключ без построения отчёта:

```powershell
python main.py --check-dgis-key
```

Для проверки конкретного адреса и фактического региона:

```powershell
python main.py --check-dgis-key --address "Пермь, Ленина 50"
```

Проверка делает три коротких запроса:

1. `region/resolve` — определяет регион через геокодер 2GIS и поле `items.region_id`.
2. `catalog/rubric/list` — проверяет доступ к официальному региональному каталогу рубрик.
3. `3.0/items` — проверяет доступ к Places API на небольшой выборке объектов.

`[OK]` означает, что соответствующий endpoint доступен и вернул данные.

`[EMPTY] places/items` означает, что Places API доступен, но тестовый запрос по выбранной рубрике и радиусу не нашёл объектов. Это не ошибка ключа и не блокер для анализа. Основной анализ использует набор рубрик из конфигурации и больший контекст локации.

`[FAIL]` с `403 apiKeyIsBlocked` означает, что активный ключ не подходит для анализа POI, парковок, домов и инфраструктуры. Такой запуск отчёта будет недостоверен, поэтому основной pipeline останавливается до расчёта.

Приоритет источников ключа:

1. Переменная окружения `DGIS_API_KEY`.
2. `.env` рядом с exe.
3. `.env` в корне проекта при запуске из исходников.
4. Встроенный ключ, если сборка была сделана с embedded secret.

Важно: если в PowerShell уже задан `$env:DGIS_API_KEY`, он имеет приоритет над `.env`. После изменения `.env` нужно либо перезапустить PowerShell, либо явно сбросить переменную:

```powershell
Remove-Item Env:DGIS_API_KEY -ErrorAction SilentlyContinue
```

Затем можно задать новый ключ:

```powershell
$env:DGIS_API_KEY="новый_ключ"
python main.py --check-dgis-key --address "Пермь, Ленина 50"
```
