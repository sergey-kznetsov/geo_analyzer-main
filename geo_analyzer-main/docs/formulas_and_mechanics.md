# Формулы и механики расчёта (для защиты проекта)

Документ описывает все расчётные механики Geo Analyzer: формулы, веса,
константы, источники данных и допущения. Значения соответствуют коду
(`src/geo_analyzer/...`) по состоянию на ревизию 2026-06-15.

Общая шкала большинства метрик — **0–10** (с обрезкой `clip(0,10)`); часть
индексов дублируется в шкале 0–100.

---

## 0. Поток данных и зоны доступности

1. **Локация.** Адрес геокодируется через 2GIS Geocoder → широта/долгота; либо
   принимаются готовые координаты (`--lat/--lon`).
2. **Изохроны.** Пешие изохроны 5/10/15 минут берутся из 2GIS Isochrone API.
   Накопительные зоны 2GIS преобразуются в **непересекающиеся кольца**
   `0–5`, `5–10`, `10–15` (вычитание меньшей изохроны из большей) — чтобы один
   объект не считался трижды.
3. **POI.** Объекты грузятся из 2GIS Catalog по рубрикам/запросам из
   `config.yaml`, дедуплицируются и привязываются к кольцам пространственным
   соединением.
4. **Метрики.** На основе POI×зоны считаются качество среды, доступность,
   сетевые метрики, притяжение, антидрайверы, парковки.
5. **Бенчмарк и отчёт.** Сравнение с районными порогами и городским snapshot,
   экспорт в Excel + текстовое саммари + визуализации.

**Зональные коэффициенты** (вес объекта по удалённости) используются почти
везде; базовый набор: `0–5 мин → 1.0`, `5–10 мин → ~0.62–0.82`,
`10–15 мин → ~0.15–0.7` (точные значения зависят от метрики, см. ниже).

## 1. Парковки (`parking/supply.py`)

### 1.1 Главный принцип
Считаем только парковочные места, **реально доступные жителям дома**, а не
посетителям ТЦ или сотрудникам офисов/производств.

### 1.2 Итоговая формула
Для каждой зоны (`0–5`, `5–10`, `Итого до 10 минут`):

```
потребность (машиномест) = Квартир_в_зоне × 0.8        # 0.8 = коэф. автовладения
потенциал = Парковочных_мест / потребность × 10
Оценка_из_10 = clip(потенциал, 0, 10)
```

- `0.8` — `CAR_OWNERSHIP_COEF` (среднее число авто на квартиру).
- `Парковочных_мест` — сумма мест только по **учитываемым** парковкам зоны.
- Если квартир в зоне нет (`потребность = 0`) — оценка не считается (`NaN`).
- Класс: `≥8 Высокий`, `4–8 Средний`, `<4 Низкий`.

### 1.3 Что учитывается (плюсуется) и что минусуется
Классификатор `_classify_parking_type` идёт по приоритету (первое совпадение
выигрывает):

| № | Условие | Результат |
|---|---|---|
| 1 | Служебные/корпоративные («для сотрудников», «по пропуску») | **Исключить** |
| 2 | Подземные (`level_count<0`, `purpose`/текст «подземн») | **Исключить** |
| 3 | Жилые/дворовые (ЖК, для жильцов, двор) | **Учесть** |
| 4 | Привязка к нежилому объекту (ТЦ, офис, производство, вокзал, досуг) | **Исключить** (для посетителей/сотрудников) |
| 5 | Общественные (городские/уличные/муниципальные) | **Учесть** |
| 6 | Платные общедоступные (`is_paid=true` или текст) | **Учесть** |
| 7 | Возможность купить/арендовать машиноместо | **Учесть** |
| 8 | Коммерческие/частные **без** продажи мест | **Исключить** |
| 9 | Закрытые/приватные **без** продажи мест | **Исключить** |
| 10 | Бесплатные / многоуровневые без нежилой привязки | **Учесть** |
| 11 | Открытая/наземная без привязки | **Учесть** |

Приоритет нежилого владельца (п.4) **выше** возможности аренды: место «при
ТЦ/офисе» жильцам недоступно, даже если формально сдаётся.

### 1.4 Источники данных (2GIS) и приоритет вместимости
Парковки берутся прежде всего **структурно** через `type=parking` со
спец-полями 2GIS. Число мест (`Парковочных_мест`) определяется по приоритету:

1. `items.capacity` — структурное поле 2GIS (факт);
2. число мест из `attribute_groups` (пары имя/значение);
3. число из текста («на N мест», «N машино-мест»);
4. **оценка по площади контура** `geometry.hull`: `места = площадь_м² / 25`
   (`AREA_PER_PARKING_SPACE_M2 = 25`, минимальная площадь 20 м²);
5. **оценка по типу** (фолбэк): многоуровневая 220, подземная 80, платная/
   автостоянка 55, паркинг 60, парковка 35, иначе 25.

Платность берётся из `items.is_paid`, подземность — из `items.level_count<0`
или `purpose`. Текстовый поиск, добор по рубрике парковок и оценки остаются
фолбэком, когда структурных полей нет.

### 1.5 Сторона спроса (квартиры)
`Квартир_в_зоне` = сумма квартир по жилым домам зоны. Число квартир в доме:

1. прямое поле/`attribute_groups` 2GIS (факт);
2. `подъезды × квартир_в_подъезде`;
3. из текста («N квартир»);
4. оценка по этажности: `этажи × подъезды × 4` (мин. 12);
5. фолбэк: типовой дом `9 × 2 × 4 = 72` квартиры
   (`DEFAULT_APARTMENTS_PER_BUILDING`).

В отчёте помечается, какие значения фактические, а какие оценочные
(колонки «Оценочное_значение», «Метод_расчёта»).

### 1.6 Автономность
Подсистема берёт от анализатора только координаты и изохроны; жилые дома и
парковки тянет из 2GIS сама, кеширует (`cache/parking_supply`), уважает
`--no-api`/refresh-cache.

## 2. Конкурентный анализ (`competition/competitors.py`)

Оценивает насыщенность рынка жильём вокруг точки (конкуренты-застройщики).

### 2.1 Сбор данных
- собственный запрос 2GIS по новым/строящимся ЖК и новостройкам (текстовый
  поиск + добор по рубрике жилых комплексов);
- переиспользование кеша мест `cache/places/*.parquet`;
- контекст из benchmark-снапшотов города.

### 2.2 Фильтрация «это конкурент?»
`_is_complex`:
- **институциональные** объекты исключаются всегда (колледж, училище,
  общежитие, больница, университет, санаторий и т.п.) — даже если в названии
  есть «жилой комплекс»;
- негативные маркеры (магазин/ремонт/автосервис/парковка) исключают, если нет
  позитивных;
- позитивные маркеры: «жилой комплекс», «ЖК», «новострой», «застройщик»,
  «квартиры от застройщика», «клубный дом», «апарт».

### 2.3 Признаки объекта
- **Новый/строящийся** (`_is_new_or_recent`): маркеры «строящ/сдача/очередь/
  котлован/старт продаж/бронирование», «новострой», либо
  `год_постройки ≥ текущий_год − 3` (`RECENT_BUILT_YEARS = 3`).
- **Квартир_оценка**: из `attribute_groups` («количество квартир», без
  ценовых атрибутов) → из текста → фолбэк `150`
  (`DEFAULT_UNITS_PER_COMPLEX`).
- **Застройщик**: из атрибутов «застройщик/девелопер».
- **Расстояние** до точки: плоское приближение
  `d = hypot(Δlon·M·cos(lat), Δlat·M)`, где `M = 111 320 м/градус`.
- **Зона расстояния**: `0–500`, `500–1000`, `1000+` м.

### 2.4 Агрегаты (сводка)
По каждой зоне расстояния и «Итого»:
- `Конкурентов_всего` = число ЖК;
- `Новых_или_строящихся` = число новых/строящихся;
- `Квартир_у_конкурентов` = сумма `Квартир_оценка`;
- `Застройщиков` = число уникальных застройщиков.

Плюс сводка по застройщикам (объектов, квартир, новых) и текстовый вывод.
Отдельный пайплайн `--competition` сохраняет результат в Excel + JSON.

## 3. Качество среды (`metrics/environment_quality.py`)

### 3.1 Взвешенная функциональная оценка
Для функциональной группы:

```
weighted_value = Σ ( Количество × criticality_score × zone_coef )
score = clip( weighted_value / normalization_target × 10, 0, 10 )
```

- `zone_coef` зависит от группы и зоны (примеры: daily `1.0/0.7/0.2`,
  education `1.0/0.75/0.35`, medicine `0.85/0.75/0.55`, leisure
  `0.85/1.0/0.7`, transport `1.0/0.65/0.25`, secondary `0.25/0.2/0.1`);
- `normalization_target` (целевой объём): daily 95, education 75, medicine 55,
  sport 35, leisure 85, secondary 50.

### 3.2 Базовые компоненты
- `daily_score`, `education_score`, `medicine_score`, `sport_score`,
  `leisure_score`, `secondary_score` — по формуле 3.1;
- `diversity_score = clip(уникальных_категорий / 12 × 10, 0, 10)`;
- `entropy_score = clip(энтропия_Шеннона / 2.3 × 10, 0, 10)`;
- `transport_score` — из сетевых метрик (см. §5), фолбэк по остановкам/авто;
- `parking_score` — парковочный потенциал «Итого до 10 минут» (§1);
- штраф антидрайверов: `anti_penalty = clip(penalty / 12, 0, 3)`.

### 3.3 Итоговые показатели (веса)
```
Инфраструктурная насыщенность =
  daily·0.45 + medicine·0.18 + education·0.12 + sport·0.08
  + secondary·0.04 + diversity·0.13 − anti·0.20

Семейная пригодность =
  education·0.42 + daily·0.22 + medicine·0.16 + transport·0.14
  + leisure·0.06 − anti·0.25

Досуг и рекреация =
  leisure·0.50 + sport·0.18 + daily·0.10 + transport·0.10
  + diversity·0.12 − anti·0.15

Функциональное разнообразие =
  diversity·0.38 + entropy·0.30 + daily·0.16 + leisure·0.10 + medicine·0.06

Проницаемость сети =
  transport·0.48 + daily·0.16 + leisure·0.12 + diversity·0.14
  + entropy·0.10 − anti·0.10

Энтропия функций (итог) = clip(entropy − anti, 0, 10)
```
Все результаты обрезаются в `[0,10]`. Шкала классов: `0–3` низко, `3–6`
средне, `6–8` хорошо, `8–10` сильно.

## 4. Доступность (`metrics/accessibility.py`)

Снимок по непересекающимся зонам 5/10/15 мин.

```
Пешая_доступность = clip( min(POI/40·5.5, 5.5)
                        + min(категорий/10·3.5, 3.5)
                        + min(точек_притяжения/3·1.0, 1.0), 0, 10 )

Остановочная_доступность: 0→0, 1→3, 2→5, 3→7, 4–5→8.5, ≥6→10

Авто_доступность (по времени до центра, мин):
  ≤5→10, ≤10→8.5, ≤15→7, ≤20→5.5, ≤30→4, ≤45→2.5, иначе 1

Итоговая = (Пешая·0.56 + Остановочная·0.24 + Авто·0.20) × zone_weight
zone_weight: 0–5→1.0, 5–10→0.82, 10–15→0.62
```
Остановки и городские точки притяжения определяются по рубрикам/ключевым словам
и дедуплицируются. Авто-время до центра берётся из 2GIS Routing.

## 5. Сетевые метрики (`metrics/centrality.py`)

Нормировки к 0–10:
```
POI 5 мин   = clip(POI5 / 35 · 10)
POI 10 мин  = clip(POI10 / 95 · 10)
POI 15 мин  = clip(POI15 / 150 · 10)
Разнообразие= clip(категорий10 / 20 · 10)
Покрытие    = clip(площадь_10мин_км² / 2.5 · 10)
Плотность   = clip((POI10/площадь) / 180 · 10)

Индекс транспортной доступности =
  POI10·0.42 + Разнообразие·0.30 + Плотность·0.18 + Покрытие·0.10
```
Площадь изохроны считается в проекции EPSG:3857.

## 6. Индекс притяжения (`metrics/attraction.py`)

Балл объекта (0–10):
```
балл = clip( category_weight × distance_weight × rating_weight × reviews_weight × 10, 0, 10)
```
- `category_weight`: городской масштаб 1.0, поддерживающий 0.55, фон 0.0
  (парикмахерские/ПВЗ/аптеки/банки/игровые автоматы и т.п. блокируются);
- `distance_weight`: `1.0 / 0.82 / 0.62 / 0.15` (по зонам, иначе 0.35);
- `rating_weight = 0.76 + (rating/5)·0.24` (0 при отсутствии → 0.76);
- `reviews_weight = min(1.18, 0.78 + ln(1+reviews)/12)`.

Сводный индекс (0–100):
```
индекс = clip( mean(top20)·2.8 + mean(top10_городских)·2.9
             + транспорт10·1.2 + плотность10·0.9
             + min(категорий/20·10,10)·0.9 + city_bonus, 0, 100)
city_bonus = min(2.2, число_городских_точек_в_10мин · 1.1)
```
Классы: `0–35` слабый, `36–60` средний, `61–80` хороший, `81–100` сильный.

## 7. Антидрайверы (`metrics/anti_driver_score.py`)

Негативное соседство ищется по 2GIS POI + семантическим запросам. Каждому типу
присвоен **штраф (severity)**:

| Тип | Штраф |
|---|---|
| Промышленность, кладбище, свалка | 10 |
| Железная дорога, ТЭЦ/энергетика | 9 |
| АЗС | 8 |
| Шумная магистраль | 7 |
| Трамвайные пути | 6 |
| Гаражи и автостоянки | 5 |

```
Суммарный_штраф = Σ severity по найденным объектам
Итоговый штраф (penalty, 0–10) = min(Суммарный_штраф / 25, 10)
```
Адрес из текста исключён намеренно (чтобы «Железнодорожная улица» не стала
антидрайвером); остановки/парки/школы/ТЦ исключаются из срабатываний.
Этот `penalty` входит в качество среды как `anti = clip(penalty/12, 0, 3)`.

## 8. Разнообразие — энтропия Шеннона (`metrics/diversity.py`)

```
H = − Σ ( p_i · ln p_i ),   p_i = count_i / Σ count
```
Нормируется к 0–10 делением на 2.3 (≈ ln(10)).

## 9. Бенчмаркинг (`metrics/benchmarking.py`)

- **Районный benchmark** — пороговый из `config.yaml` (`low/medium/high`),
  даёт класс «Низкий/Средний/Хороший/Сильный уровень».
- **Городской benchmark** — сохранённый snapshot города (первый успешный анализ
  или ручное обновление). Сравнение:
  ```
  delta = Фактическая_оценка − Бенчмарк_города_из_10
  ≥ +1.0 → «Выше города»; ≤ −1.0 → «Ниже города»; иначе «На уровне города»
  ```

## 10. Сравнение локаций (`pipeline/comparison_pipeline.py`)

Полный анализ для A и B, извлечение метрик, попарное сравнение:
- победитель по метрике: больше = лучше (для времени до центра/антидрайверов —
  меньше = лучше); разница < 0.001 → «Паритет»;
- итоговый балл локации `= доля побед × 10` (паритет = 0.5 победы);
- при разных городах добавляется победитель по отклонению от городского
  benchmark;
- экспорт: `comparison.xlsx` (Сравнение / Метрики / Преимущества A/B / Саммари /
  Сравнение с benchmark), `comparison_summary.txt`, графики и карта.

## 11. Источники данных: кеш + бенч + сеть

Все загрузочные модули (places, parking, competition, anti-drivers, routing,
isochrones) работают по единой схеме:
1. при `use_cache` и без `refresh-cache` — читают локальный кеш
   (`data/cache/...`);
2. при `--no-api` без кеша — возвращают пусто/ошибку (сеть не дёргается);
3. иначе — запрос к 2GIS и сохранение в кеш.

Городской benchmark хранится в `data/benchmarks`. Конкурентный анализ
дополнительно читает кеш мест и benchmark-снапшоты.

## 12. Ключевые константы и допущения (для защиты)

| Константа | Значение | Где |
|---|---|---|
| Коэффициент автовладения | 0.8 | парковки |
| Площадь на 1 машино-место | 25 м² | оценка вместимости |
| Типовой дом (квартир) | 72 (9×2×4) | оценка спроса |
| «Недавно построен» | ≤ 3 лет | конкуренты |
| Квартир в новостройке (фолбэк) | 150 | конкуренты |
| Скорость пешехода | 4.8 км/ч | изохроны (config) |
| Радиус POI | 1200 м | config |
| Зоны изохрон | 5/10/15 мин | config |
| Нормировка штрафа антидрайверов | /25 | антидрайверы |

**Принцип честности отчёта:** каждое оценочное (не подтверждённое 2GIS)
значение помечается в отчёте (колонки «Оценочное_значение», «Метод_расчёта»,
«Метод_расчёта_мест»), чтобы было видно, где факт, а где расчётная оценка.
