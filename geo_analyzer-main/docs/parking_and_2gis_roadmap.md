# Парковки и развитие на базе 2GIS API (для застройщика)

## 1. Механика парковок (как считается сейчас)

Подсистема парковок вынесена в отдельный пакет `geo_analyzer.parking` и
работает автономно от основного анализатора:

- **что берётся от анализатора:** координаты точки (`latitude`/`longitude`),
  радиус и геометрия изохрон (зоны пешей доступности 5/10 минут);
- **что подсистема тянет сама:** жилые дома и парковочные объекты из 2GIS
  Catalog API (`3.0/items`). Парковки берутся прежде всего **структурно** —
  запросом `type=parking` со специальными полями 2GIS: `items.capacity`
  (число мест), `items.is_paid` (платность), `items.purpose` (назначение),
  `items.level_count` (этажность; `<0` → подземная). Текстовый поиск, добор по
  рубрике парковок, чтение `attribute_groups` и оценка по площади контура
  (`geometry.hull`) остаются фолбэком, когда структурных полей нет.

### Формула обеспеченности

```
потребность   = квартиры_в_зоне × коэффициент_владения_авто (0.8)
потенциал/10  = clip( парковочные_места / потребность × 10, 0..10 )
```

- **квартиры** считаются по зонам 0–5 и 5–10 минут (из атрибутов 2GIS, либо
  оценочно: этажи × подъезды × квартир на этаж);
- **места** суммируются только по парковкам, которые реально доступны жильцам.

### Что плюсуется, а что минусуется

Главный принцип: **парковка должна быть доступна жителям дома**, а не
посетителям ТЦ или сотрудникам офиса/производства.

| Учитывается (`+`) | Исключается (`−`) |
|---|---|
| Общественные / уличные / муниципальные | Привязанные к нежилым объектам — для посетителей/сотрудников (ТЦ, офис, завод, вокзал, досуг) |
| Платные общедоступные | Служебные / корпоративные (для сотрудников, по пропуску) |
| Коммерческие стоянки **с продажей или арендой машиноместа** | **Подземные паркинги** (как правило приватные места при здании) |
| Жилые/дворовые (ЖК, для жильцов) | Закрытые/приватные и коммерческие **без** продажи/аренды мест |
| Открытые/наземные без нежилой привязки | |

Приоритет признаков: служебные → **подземные** → жилые (ЖК/двор) → нежилой
владелец (ТЦ/офис/производство) → общественные → платные → продажа/аренда
места → коммерческие без продажи → закрытые без продажи → открытые наземные.
Привязка к нежилому объекту важнее возможности аренды: место «при ТЦ/офисе»
жильцам недоступно, даже если формально сдаётся.

Прозрачность вынесена в отчёт: в детализации — колонка `Можно_купить_место`,
причина учёта/исключения и логика фильтрации; в сводке — `Мест_по_покупке_аренде`
и `Исключено_закрытых_и_коммерческих`.

## 2. Идеи улучшений на базе 2GIS API (приоритет для застройщика)

Застройщику важны спрос, конкуренция, доступность и инфраструктурные риски
площадки. 2GIS Catalog/Routing/Geocoder дают для этого данные напрямую.

### Высокий приоритет
1. **Цены и тариф парковок.** Из `attribute_groups` вытягивать тариф/цену
   машиноместа и помечать парковки с продажей мест ценой. Даёт застройщику
   рыночный ориентир стоимости машиноместа в районе и обоснование цены своего
   паркинга.
2. **Конкурентная застройка (ЖК и новостройки рядом).** ✅ Реализовано как
   отдельный самодостаточный модуль `geo_analyzer.competition` со своим
   пайплайном (`pipeline/competition_pipeline.py`, CLI-флаг `--competition`).
   Считает число конкурирующих ЖК/новостроек по зонам расстояния, долю новых/
   строящихся, оценку числа квартир, сводку по застройщикам; собирает данные из
   2GIS + кеша мест + benchmark-снапшотов. Дальше можно добавить цены квартир и
   темпы продаж.
3. **Баланс спроса и предложения по парковкам в динамике зон.** Уже считается
   до 10 минут; добавить разрез «во дворе (0–5)» против «общественные (5–10)»,
   чтобы видеть дефицит именно дворовых мест — типичная боль покупателя.

### Средний приоритет
4. **Транспортная доступность через Routing API.** Время до центра города,
   ключевых работных кластеров (БЦ) и до ближайшего въезда на магистраль —
   на авто и на общественном транспорте. Сильный маркетинговый аргумент.
5. **Социальная инфраструктура с ёмкостью.** Не просто «школа есть», а
   расстояние и загрузка: школы/сады/поликлиники из 2GIS с атрибутами,
   расчёт «мест на жителя» в зоне — критерий для семейного покупателя.
6. **Расписания и «живость» места.** Поле `schedule` уже запрашивается:
   доля круглосуточных объектов, плотность точек притяжения по времени —
   индикатор оживлённости и безопасности района.

### Низкий приоритет / задел
7. **Геометрия зданий (`geometry.hull`) для 3D-контекста** конкурентов и
   собственного пятна застройки на карте/изометрии.
8. **Отзывы и рейтинги** объектов рядом (если доступны в тарифе API) — оценка
   качества окружения.
9. **Кэш-снапшоты по дате** для отслеживания изменений района между анализами
   (новые ЖК, новые парковки) — мониторинг площадки во времени.

> Реализацию п.1–3 разумно вести внутри пакета `geo_analyzer.parking` и
> симметричного пакета конкуренции/спроса, чтобы не разрастался основной
> анализатор — по тому же принципу автономности, что и парковки.
